## 1. 这个项目到底是什么？

一句话：**一个用纯 Python + NumPy 从零手写的文本生成引擎**。

它做的事情和 ChatGPT 的核心是一样的——给定一句话的上半部分，预测下一个字/词应该是什么。区别是：
- ChatGPT 有几千亿参数，这个项目只有 **100 万**参数（所以不能真的拿来用）
- ChatGPT 用 PyTorch + GPU 集群跑，这个项目在一个笔记本 CPU 上就能跑
- ChatGPT 内部是黑盒，这个项目 **每一行代码你都能看懂**

类比后端世界：这就像有人用裸 TCP  socket + 自己实现了 HTTP/1.1 协议栈。功能肯定不如 Nginx，但读完你就知道 HTTP 底层到底怎么运作的。

## 2. 先忘掉那些术语——用后端概念做类比

机器学习很多术语听起来唬人，但本质都是你日常写后端的东西换个名字：

| 机器学习的说法 | 后端里的对应概念 |
|---|---|
| 模型（Model） | 一个**函数**或**服务** |
| 参数（Parameters） | 函数的**配置项**（类似配置文件里的参数） |
| 训练（Training） | **调参**——不断试，找到让输出最好的那组配置 |
| 推理（Inference） | 用**已经调好的配置**跑一次请求 |
| 损失（Loss） | 当前输出的**错误率/误差** |
| 反向传播（Backprop） | **定位哪个参数导致了误差，需要往哪个方向调** |
| 梯度（Gradient） | 参数调整的**方向和力度** |
| 优化器（Optimizer） | 调参的**策略/算法** |
| 嵌入（Embedding） | 把某个东西映射成一个**固定长度的 ID 向量** |
| 张量（Tensor） | 就是**多维数组**（numpy array） |

接下来我们用这个"翻译表"展开讲。

## 3. 业务需求：我们要解决什么问题？

**输入**："春眠不觉"

**输出**：模型预测下一个最可能的字 → "晓"

这就是整个系统的核心功能。和后端 API 一样：有输入，有输出，只是处理逻辑不是 `if-else` 或数据库查询，而是一千多万次浮点数矩阵乘法。

那模型怎么"学"到这个答案的呢？

**训练过程类比**：
1. 给模型看 10 万首唐诗（数据集）
2. 每次遮住一个字让它猜
3. 猜对了 → 保持当前配置
4. 猜错了 → 微调内部参数，让下次猜对概率大一点
5. 重复几十万次

这和一个后端系统通过 A/B Test + 调参优化接口响应时间，本质上没有区别——都是 **通过反馈信号迭代优化内部参数**。

## 5. 数据处理：和 ETL 没区别

`download_corpus()`（L122）做的事情：

1. 从 GitHub 唐诗数据集 JSON 文件下载原始数据
2. 解析每首诗的标题、作者、正文
3. 只保留中文字符（正则过滤）
4. 拼接成一个大文本文件 `corpus.txt`
5. 如果网络不可用，用内置的 fallback 语料兜底

**后端视角**：这是一个标准的 ETL Pipeline —— Extract（拉数据）、Transform（清洗）、Load（存本地缓存）。

`build_vocab()`（L328）：
- 统计所有字出现的频率
- 取出现次数最多的 5000 个字作为词表
- 建立 `字 → 编号` 的映射字典（就是一个 HashMap）

`prepare_data()`（L346）：
- 把文本切成一段段的，每段 128 个字符（MAX_SEQ_LEN）
- 每段的输入是前 128 个字，输出是后 128 个字整体往后错一位
  - 例：输入 "春眠不觉晓"，目标输出 "眠不觉晓处"
- 这就是训练数据的构造方式：让模型看前 N 个字，猜第 N+1 个字

## 6. 模型核心：逐层拆解（重点）

### 6.1 嵌入层：把字符变成向量

```python
token_emb = params['token_embedding'][x]  # (batch, seq_len, D_MODEL)
```

**类比**：数据库里做 JOIN。

`token_embedding` 是一个 `(5000, 64)` 的矩阵——你可以理解为一个**字典/HashMap**：
- Key：字的编号（0 ~ 4999）
- Value：一个 64 维的浮点数向量

当输入是字编号 `[3, 150, 8]` 时，embedding 就是从这个表中"查"出 3 个 64 维向量。

**为什么需要这样做？**
因为数学运算只能对数字做，不能对"字"做。我们需要把每个字映射到一个**数值空间**中，使得语义相近的字在这个空间里距离也近（比如"春"和"秋"都是季节，它们的向量会比较接近）。

但这个"语义接近"不是我们手动编码的，而是训练过程中自动涌现出来的——就像你不需要告诉模型"春和秋都是季节"，看了足够多文本后它自己会学到。

还有一个 `position_embedding`（位置嵌入），也是查表，但是按**位置**（第 0 个位置、第 1 个位置……第 127 个位置）查。因为模型本身不理解顺序——你不告诉它"春眠"和"眠春"位置不同，它就认为是一样的。

```python
h = token_emb + pos_emb  # 语义信息 + 位置信息
```

这个 `+` 就是把两个向量逐元素相加，得到同时包含"是什么字"和"在哪个位置"信息的表示。

---

### 6.2 Transformer Block：系统的核心中间件

模型有 4 个 Transformer Block 串联，数据从头流速经过每一层。每一层的结构完全一样（就像 4 个相同的中间件 pipeline stage）。

每一层内部有两步（Pre-LN 架构——你可以记成"先标准化再处理"）：

```
输入 x
  │
  ├──→ LayerNorm(x) ──→ Multi-Head Attention ──→ attn_out
  │                                                       │
  │              x + attn_out ←──────────────────────────┘   ← 残差连接
  │              │
  ├──→ LayerNorm(上面结果) ──→ FeedForward ──→ ff_out
  │                                                    │
  │             上面结果 + ff_out ←────────────────────┘   ← 残差连接
  │              │
  └── 输出（传给下一层）
```

#### LayerNorm：数据归一化

类比后端的**请求预处理**——把数据标准化到统一尺度，避免极端值影响后续处理。

对一维向量（64 个数），计算均值和方差，然后：
```
normalized = (x - mean) / sqrt(variance + eps)
output = gamma * normalized + beta
```

`gamma` 和 `beta` 是可学习参数——模型自己学习**应该缩放到什么范围**。

#### 多头注意力（Multi-Head Attention）：核心中的核心

这是 Transformer 名字里 "Transform" 的由来。

**一句话解释**：对于序列中的每个位置，注意力机制让它"看一眼"序列中其他位置的词，判断哪些跟当前预测相关，然后按相关性加权汇总信息。

**举个例子**："春眠不觉晓，处处闻啼鸟"

当模型在预测"晓"后面的字时：
- 它需要回头看前面的字来判断上下文
- "处"字跟"春眠不觉晓"语义相关度高（都在描写清晨的景象）
- 注意力机制就是自动学习"对于当前预测位置，应该多关注哪些其他位置"

**"多头"是什么意思？**
就像你审读一篇文章时，一个人关注情节线，另一个人关注人物关系，第三个人关注修辞手法——多个"注意力"同时工作，每个关注不同维度的信息。这里是 4 个头（`N_HEADS`）。

**因果掩码（Causal Mask）**：这是 GPT 和其他模型（如 BERT）的关键区别。

GPT 是**自回归**的——一个字一个字地生成，不能偷看后面的字。所以当我们计算注意力时，要把"未来位置"的信息屏蔽掉：

```
位置:    0    1    2    3    4
     ┌─────────────────────────┐
  0  │  ✓   ✗   ✗   ✗   ✗     │  位置 0 只能看自己
  1  │  ✓   ✓   ✗   ✗   ✗     │  位置 1 能看 0,1
  2  │  ✓   ✓   ✓   ✗   ✗     │  位置 2 能看 0,1,2
  3  │  ✓   ✓   ✓   ✓   ✗     │  位置 3 能看 0,1,2,3
  4  │  ✓   ✓   ✓   ✓   ✓     │  位置 4 能看全部
     └─────────────────────────┘
```

代码里实现很简单：生成一个上三角矩阵，非零位置填 `-1e9`（负无穷），加到注意力分数上，softmax 之后这些位置就变成了 0。

#### 前馈网络（FeedForward）

```python
h = x @ W1 + b1      # 64 → 256（升维）
h = ReLU(h)          # 激活函数：把所有负数归零
output = h @ W2 + b2 # 256 → 64（降维回来）
```

就是个标准的两层全连接网络：先升维再降维。

**类比后端的意义**：注意力层负责"收集信息"（跟其他位置交互），FFN 负责"消化信息"（对收集到的信息做非线性变换和抽象）。就像一个 worker 从消息队列取消息（注意力），然后处理业务逻辑（FFN）。

#### 残差连接（Residual Connection）

```python
x = x + attn_out
```

输入直接加到输出上，绕过中间处理。

**类比后端**：类似于**事件溯源（Event Sourcing）**——你保留原始事件日志（x），同时在上面叠加处理后的结果（attn_out）。

**为什么需要？**
没有残差连接时，深层网络训练不动——梯度在反向传播时指数级衰减/消失（类似微服务调用链太长，最后一层根本收不到第一层的反馈信号）。残差连接保证梯度有一条"高速公路"直达底层。

### 6.3 模型参数总览

`init_model()`（L396）定义了所有"配置项"：

| 参数名 | Shape | 作用 |
|---|---|---|
| `token_embedding` | (5000, 64) | 字符→向量查表 |
| `position_embedding` | (128, 64) | 位置→向量查表 |
| `block_{i}_W_q/k/v/o` | (64, 64) × 4层 | 注意力投影矩阵 |
| `block_{i}_ln1/2_gamma/beta` | (64,) × 4层 | LayerNorm 缩放/偏移 |
| `block_{i}_W1/b1/W2/b2` | (64,256)/(256,)/(256,64)/(64,) × 4层 | FFN 两层的权重和偏置 |
| `ln_f_gamma/beta` | (64,) | 最终归一化 |

总计约 1.1M 个浮点数参数——都是很小的矩阵。

**权重共享**（Weight Tying）是一个巧妙的设计：
```python
logits = h @ params['token_embedding'].T  # L770
```
模型输出的线性层，**复用了输入的 embedding 矩阵**（转置使用）。省参数，效果还好。

## 7. 训练循环：一个超级慢的调参过程

`train_step()`（L929）的逻辑：

```
前向传播 → 算损失 → 反向传播 → 梯度裁剪 → 参数更新
```

这 5 步跟"调优一个后端系统"的直觉完全一样：

### 7.1 前向传播（Forward Pass）

就是输入数据，跑一遍模型的所有计算，得到预测结果。跟调用一个正常的 API 请求没区别，只不过这个 API 内部跑了上百万次浮点乘法。

### 7.2 损失函数（Loss Function）

模型预测的概率分布 vs 真实答案（one-hot），算交叉熵。

**后端类比**：这就是你的**监控指标**——接口 P99 延迟、错误率。损失函数越低，说明模型预测越准。

### 7.3 反向传播（Backpropagation）

这是整个算法最精妙的部分。

**核心思想**：用微积分的链式法则，从最终输出端（损失）往输入端反推，计算出"每个参数对最终误差贡献了多少"。

**类比后端**：

想象你有一个 20 层调用的微服务调用链：
```
Client → Gateway → Auth → RateLimit → ServiceA → ServiceB → DB
```

DB 查询慢了 10 倍（相当于"损失"高了）。你需要定位是哪里出了问题、每个服务"应该为这次慢查询负多大责任"——这就是分布式 Trace 在做的事。

反向传播本质就是**数学化的分布式 Trace**：
- 每个参数 = 一个微服务
- 梯度 = 该参数对误差的"责任占比"
- 链式法则 = 按调用链反推责任

具体到代码中，就是按前向传播的**逆序**，逐层求导：

```
cross_entropy_backward()     # 损失函数对 logits 的梯度
        ↓
gpt_forward 的逆操作：        # logits 对最后一层输出的梯度
        ↓
layer_norm_backward()        # 对 LayerNorm 输入的梯度
        ↓
transformer_block_backward() # 4 层 Transformer 逆序反传
        ↓
multi_head_attention_backward() # 注意力内部的反传
        ↓
此时梯度已经回到最开始的 embedding 层面
```

每一层反传都是**高中数学的链式法则**的高维推广，没有魔法。

### 7.4 梯度裁剪

```python
grads = clip_gradients(grads, max_norm=1.0)  # L989
```

**为什么？**：当梯度太大时，一次参数更新就会"矫枉过正"——就像微服务里某个参数被调了一下直接就从 1 改成 10000，系统直接炸了。

**做法**：计算所有梯度的总范数，如果超过阈值，等比缩小。

**类比**：限流/熔断。不管请求多大，每秒最多处理 N 个。

### 7.5 Adam 优化器更新参数

Adam 是目前最常用的优化算法。它比普通梯度下降聪明在于：
- **动量（Momentum）**：记住上一次更新方向，避免在坑洼处震荡（类似惯性）
- **自适应学习率**：不同参数用不同的学习率，更新频繁的参数步子小一点，安静的参数步子大一点

学习率调度：
- **Warmup 阶段**（前 500 步）：从 0 线性增到 3e-3
  - 类比：系统刚上线先小流量灰度，再逐渐放量
- **Cosine 衰减阶段**：从 3e-3 按余弦曲线缓慢降到接近 0
  - 类比：系统稳定后逐渐减小变更幅度，趋于收敛

## 8. 文本生成：对外暴露的 API

`generate()`（L1059）是系统对外暴露的"接口"：

```python
def generate(prompt, params, char2idx, idx2char, max_new_tokens=100, temperature=0.8, top_k=50):
```

**逐字自回归**：每次把当前所有文字送入模型，预测下一个字，拼到末尾，重复。

**Temperature 参数**（temperature）：
- 高（如 1.5）：更随机、更有创意（输出更像探索性测试）
- 低（如 0.2）：更确定、更安全（输出更像幂等的稳定服务）

**Top-k 采样**：
- 不是每次都在 5000 个字里选
- 而是在概率最高的前 k 个（k=50）里选
- 类比：排行榜 Top-N，不考虑长尾

**类比**：这样的生成流程就像是一个 while 循环的 RPC 调用：
```
tokens = prompt
while len(tokens) < max_len:
    next_token = model.predict(tokens)  # RPC 调用
    tokens.append(next_token)
    if next_token == EOS:
        break
return tokens
```

## 12. FAQ

**Q：这 1.1M 参数到底"记住"了什么？**
A：不是"记住"，是"学会"。就像一个有经验的后端工程师不是记住了每个 API 的文档，而是通过大量 case 培养了直觉。参数编码的是语言的模式/规律，不是具体的文本。

**Q：和 PyTorch 版本有什么区别？**
A：PyTorch 帮你自动做了反向传播（autograd），你只需要写前向。这个项目是手推了所有反传公式。就像手写 SQL vs 用 ORM 的区别——手写慢但你能控制一切。

**Q：为什么并行度看起来不高？**
A：当前实现是单样本流水线，GPU 版本（CuPy）可以加速矩阵乘法。如果要真正的并行训练，需要像 PyTorch 那样写 DataParallel / DistributedDataParallel，那是另一个话题。

**Q：这个模型能做什么？**
A：能生成看起来像唐诗的文本。但因为参数太少、语料有限，质量不高。核心价值在于教学——让你理解 ChatGPT 的底层运作逻辑。

---

> **总结**：Transformer 不神秘。它就是一个用多维数组作为数据结构、矩阵乘法作为计算原语、链式法则作为反馈机制的参数化函数。理解了"数据流"和"参数更新"两个维度，你就理解了 90% 的深度学习。
